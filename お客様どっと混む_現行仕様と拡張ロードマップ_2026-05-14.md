# お客様どっと混む 現行仕様と拡張ロードマップ

作成日: 2026-05-14  
対象サービス: お客様どっと混む / okyakusa-ma.com  
現行位置づけ: β版 / 那須・栃木の小規模店舗向け無料診断MVP

---

## 1. このサービスの定義

「お客様どっと混む」は、小規模店舗が検索・Google Maps・口コミ・SNS・AI検索時代において、単に多くの人を集めるのではなく、**本来その店と相性の良いお客様に見つけてもらい、安心して選ばれ、来店してもらう状態**をつくるための診断・レポート・改善支援サービスである。

名称には以下の意味を重ねている。

- お客様.com
- お客様どっと混む
- 本来来てほしいお客様でお店が混む

現行MVPの価値は「分析ツール」ではなく、店舗の魅力・不安・見え方を、店主が理解しやすい言葉と点数に翻訳することにある。

---

## 2. 現行アーキテクチャ

現行は、重いSaaS基盤を避けた軽量MVP構成である。

```text
ユーザー
  ↓
okyakusa-ma.com トップページ
  ↓
Vercel Serverless API
  ↓
Google Places API / Gemini API
  ↓
Google Apps Script
  ↓
Google Sheets
  ↓
/r/view/?id={request_id} 診断結果ページ
```

### 使用している主要構成

- フロントエンド: 静的HTML / CSS / JavaScript
- ホスティング: Vercel
- ソース管理: GitHub
- 受付・診断API: Vercel Serverless Functions
- データ保存: Google Sheets
- Sheets連携: Google Apps Script
- 店舗情報取得: Google Places API
- AI診断生成: Gemini API
- 画像診断: Gemini Vision相当のマルチモーダル入力
- 結果共有: URL共有 / LINE / メール / コピー

### 現時点で意図的に作っていないもの

- ログイン機能
- 会員管理
- Stripe決済
- Supabase等の本格DB
- 管理画面
- SNS自動投稿
- Google Business Profile自動更新
- Search Console連携
- SERP順位取得
- 本格的なSaaSダッシュボード

## 3. トップページ仕様

対象ファイル: `index.html`

トップページは、Googleやaramakijake.jp的な「ひとつの診断窓から始まる」体験を意識した構成である。

### 入力フォーム

#### 必須項目

- 店舗名
- エリア
- 業種
- 申込者名
- メールアドレス

#### 任意項目

- Google Maps URL
- 公式サイトURL
- Instagram URL
- YouTube URL
- 来てほしいお客様・今困っていること

#### 削除済み・現時点では入れていない項目

- X URL
- TikTok URL
- 検索キーワード入力
- 検索順位診断入力

上記は、トップページが重くなりすぎることと、現時点で検索順位を正確に出すAPI設計が未確定であるため、一旦外している。

### URLの調べ方ヘルプ

フォーム内に「URLの調べ方」導線を設置している。

対象:

- Google Maps URL
- Instagram URL
- YouTube URL
- Xの場合
- TikTokの場合
- Facebookの場合

X / TikTok / Facebookは現時点では入力欄を持たず、必要に応じてInstagram欄または備考的に案内する想定。

### 送信時の流れ

1. 店舗情報を入力
2. 「無料で診断する」
3. `/api/diagnosis` にPOST
4. request_idを発行
5. Google Sheetsに受付保存
6. 自動診断が有効なら Places / Gemini 診断を実行
7. 完了後に結果ページへ誘導

## 12. 現時点のSEO / 検索診断の扱い

### 現行でできること

- 公式サイトURLの有無を見る
- 店舗情報と検索意図のズレをGeminiで意味的に診断する
- 地域名 + 業種 + 利用シーンの情報が足りるかを見る
- AI検索に推薦されやすい説明材料があるかを見る

### 現行でやらないこと

- Google検索順位の断定
- 「那須 ランチ」など任意キーワードの正確な順位表示
- SERPスクレイピング
- Custom Search APIによる本格SEO診断
- Search Console連携
- Google Ads Keyword Planner連携

### 2026-05-13の検索API実験で分かったこと

Google Custom Search JSON API / Programmable Search Engineを使って「お店の検索ことば診断β」を試作したが、現行プロダクトの中核にはしない判断をした。

理由:

- Programmable Search Engineは本来、自社サイト内検索や限定サイト検索向き
- 通常のGoogle検索順位とは結果が一致しにくい
- 無料枠が少ない
- 「ウェブ全体を検索」の設定がわかりにくい
- APIキー制限の変更で既存のPlaces / Gemini診断に影響が出た
- 店舗オーナーに誤解を与える可能性がある

このため、該当機能はGitでrevert済み。

今後SEOを本格化する場合は、以下のいずれかで再設計する。

- Search Console API: 契約店舗の実データを見る
- Google Ads Keyword Planner: 地域キーワード需要の目安を見る
- Gemini Search / Grounding: 検索文脈の意味的診断に使う
- SERP API: 必要な場合のみ別系統で検証する

- 地域の居場所
- ライブ感のある店

## 15. API戦略

### 現行で本番利用するAPI

- Google Places API
- Gemini API
- Apps Script Web App

### 慎重に扱うAPI

- Gemini Search / Grounding
- Gemini Vision
- Search Console API
- Google Ads Keyword Planner
- Custom Search JSON API
- SERP API

### 原則

- 本番で動いているAPIキーは触らない
- 新しいAPIは専用キーを作る
- APIキーごとに用途を分ける
- 実験は別ブランチ・別環境で行う
- 検索順位を断定しない
- 外部サイトのスクレイピングは避ける

### APIキー分離案

```text
GOOGLE_PLACES_API_KEY
GEMINI_API_KEY
GOOGLE_SCRIPT_URL
GOOGLE_SCRIPT_SECRET
GOOGLE_SEARCH_CONSOLE_CLIENT_ID
GOOGLE_SEARCH_CONSOLE_CLIENT_SECRET
GOOGLE_ADS_DEVELOPER_TOKEN
SERP_API_KEY
```

Custom Searchを再検証する場合のみ:

```text
GOOGLE_CUSTOM_SEARCH_API_KEY
GOOGLE_CUSTOM_SEARCH_CX
```

ただし、現時点では中核機能にはしない。
