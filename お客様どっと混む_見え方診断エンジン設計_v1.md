# お客様どっと混む 見え方診断エンジン設計 v1

作成日: 2026-05-19  
目的: 無料診断を「雰囲気のよいAI診断」から「根拠あるWeb見え方診断」へ進化させるための基本設計  
対象: 無料診断、深掘り診断、将来の管理画面、SNS観測、研究用データ蓄積

---

## 1. 基本方針

お客様.COMは、経営診断でも、純粋なSEO診断でも、MEO代行でもない。

定義:

> 小さなお店が、初めて見る人・Google Maps・口コミ・SNS・AI検索から、どう見えているかを診断する「Web見え方診断」

主戦場は検索順位ではなく、「初めて見る人にどう見えているか」である。

重視するもの:

- 初めて見た人が安心できるか
- 行く前の不安が消えるか
- Google Maps、口コミ、SNS、公式情報がつながって見えるか
- AIに誤解されず説明されるか
- 店主が明日何を直せばよいか分かるか

SEOは扱うが、順位獲得技術としては深追いしない。  
お客様.COMにおけるSEOは「初めて検索した人が、迷わず理解できるか」の簡易診断として扱う。

世界観カードの本格設計は別途ヒアリング後に扱う。  
本設計書では、見え方診断エンジンに集中する。

## 2. 診断エンジンの4層構造

### 2.1 観測レイヤー

AIが感想を書く前に、まず事実を集める。

観測対象:

- Google Maps情報
- 評価、口コミ件数
- 営業時間
- 駐車場
- 支払い方法
- 写真枚数
- カテゴリ
- 住所、電話、Web導線
- 公式サイト有無
- SNS URL
- 店主入力情報
- 画像がある場合は写真内容
- 将来的には、SNS投稿URL、動画内容、サムネイル、文字起こし

観測レイヤーでは評価しない。  
「何があるか」「何がないか」「何が曖昧か」を構造化する。

### 2.2 判定レイヤー

観測データを独自基準で判定する。

主要判定軸:

- Maps整備度
- 口コミ信頼度
- 入りやすさ
- 来店前不安
- AI検索で拾われやすいか
- 動画化しやすさ
- 初見客への説明力
- 保存したくなる空気感

判定は、AIの自由な感想ではなく、ルーブリックに基づいて行う。

### 2.3 根拠レイヤー

診断結果には必ず根拠を添える。

出すべき根拠:

- 何を見てそう判断したか
- どの情報が足りないか
- どの写真が足りないか
- どの文章が弱いか
- どこを直すと点が上がるか

禁止:

- 根拠なしに「弱い」「良い」と断定する
- 「改善しましょう」だけで終わる
- SEO・経営診断のように見せすぎる

### 2.4 改善指示レイヤー

店主さんが明日できる形に変換する。

出すべき改善指示:

- まず撮る写真
- Mapsに足す説明
- SNSに載せる一文
- 今月やること
- 今すぐ直すこと
- 今撮るべき動画3本

## 3. 全診断項目

### 3.1 THE お店入りやすさ診断

最重要診断。お客様.COMの核にする。

目的:

> 初めてのお客様が、行く前に不安なく「入れそう」と思えるかを見る。

観測データ:

- Maps写真枚数
- 写真に外観、入口、駐車場、店内、席、商品があるか
- 営業時間
- 駐車場
- 支払い
- 口コミ数、評価
- 公式サイト有無
- SNS URL
- 店主入力の業種、地域、悩み
- 画像アップロードがある場合は写真内容

判定軸:

| 軸 | 良い状態 | 弱い状態 |
|---|---|---|
| 外観 | 店の外から見た雰囲気が分かる | 外観写真がない |
| 入口 | どこから入るか分かる | 入口が分からない |
| 営業中感 | 営業時間、定休日、店の生きている感じがある | 営業しているか不安 |
| 駐車場 | 有無、場所、台数感が分かる | 車で行けるか不安 |
| 価格感 | 代表メニューや価格帯が分かる | 高いか安いか不明 |
| 席・店内 | 座る場所、距離感、空気が分かる | 入った後が想像できない |
| 初回導線 | 初めてでも流れが分かる | 常連向けに見える |
| 安心写真 | 外観、入口、駐車場、席、メニューが揃う | 商品写真だけに偏る |

出力:

- 入りやすさスコア
- 入りにくく見える理由
- 足りない写真
- 今すぐ撮るべき写真3枚
- Mapsに足すべき説明
- SNS投稿文

無料版:

- スコア
- 不安要素3つ
- 撮るべき写真3枚
- 短い改善コメント

有料深掘り:

- 写真ごとの不足判定
- 入口、駐車場、席、価格、初回導線の詳細改善
- Maps説明文案
- SNS投稿文3本
- 1週間の撮影プラン

### 3.2 AI de 店エゴサーチ

目的:

> AIがこの店を説明・推薦するとき、正しく魅力的に拾える材料があるかを見る。

観測データ:

- 店舗名
- 地域
- 業種
- Google Mapsカテゴリ
- 口コミ数、評価
- 公式サイト有無
- SNS URL
- Maps説明、Web導線
- 店主入力情報
- AI診断要約

判定軸:

- 何の店か一文で分かるか
- 誰におすすめか分かるか
- 地域名・用途・利用シーンが拾えるか
- 口コミやサイトから推薦理由が作れるか
- AIに誤解されにくいか

出力:

- AI検索/GEOスコア
- AIにこう見えています
- AIが拾いやすい材料
- AIが迷いそうな部分
- 足すべき説明文

### 3.3 今撮るべき動画3本

目的:

> 店舗の弱点から逆算して、今一番効く動画を3本出す。

判定ルール例:

| 弱点 | 出す動画 |
|---|---|
| 入口不安が弱い | 入口から席まで15秒動画 |
| 駐車場が不明 | 駐車場から入口まで動画 |
| 価格感が不明 | はじめての注文・代表メニュー動画 |
| 店内が見えない | 席と距離感を見せる動画 |
| 口コミ信頼が強い | 常連が選ぶ理由動画 |
| AI検索が弱い | 誰におすすめかを説明する動画 |

出力:

- 動画タイトル
- 15秒構成
- 撮る場所
- 撮る順番
- テロップ例
- 期待される効果

---

### 3.4 弱点TOP3

目的:

> 店主が最初に直すべき「見え方の穴」を3つに絞る。

優先基準:

- 来店直前の不安を増やすもの
- すぐ直せるもの
- スコアへの影響が大きいもの
- 店主が自力で改善できるもの

出力:

- 弱点名
- 根拠
- 影響
- 最初の改善行動

---

### 3.5 強みトップ3

目的:

> 店主が自覚していない「すでに使える強み」を見つける。

判定基準:

- 外から見ても伝わっているもの
- 口コミやMapsで裏付けがあるもの
- AIが推薦理由として使いやすいもの
- 動画化しやすいもの

出力:

- 強み名
- 根拠
- もっと使う方法
- SNS/Mapsでの表現例

### 3.6 今月の重点

目的:

> すべての診断結果から、今月やるべきテーマを1つに絞る。

優先順位:

1. 店舗特定が弱い
2. 入りやすさが弱い
3. Maps基本情報が弱い
4. 口コミ信頼が弱い
5. AI検索/GEOが弱い
6. 動画導線が弱い

出力:

- 今月の重点テーマ
- なぜ今月これか
- 1週間目、2週間目、3週間目、4週間目の行動

### 3.7 Google Maps / 口コミ / SEO / AI検索GEO

Google Maps:

- 店舗名、住所、カテゴリ、営業時間、電話、Web導線、写真、駐車場、支払い、口コミを診断
- 無料版では要点
- 有料版では写真指示、Maps説明文、1週間改善プラン

口コミ:

- 評価平均、口コミ件数、競合中央値、信頼材料を診断
- 将来的には返信状況や口コミ本文も対象
- 有料版では返信テンプレート、口コミ依頼文、店内POP文案

SEO:

- 軽めに扱う
- 順位ではなく、初めて検索した人が理解できるかを見る
- 公式サイト有無、地域語、業種語、アクセス、料金、FAQを確認

AI検索/GEO:

- AIが推薦文を作るときに、店の特徴を正しく拾えるかを見る
- Maps、口コミ、公式サイト、SNS、店主入力を統合
- 有料版ではAI向け店舗説明文、FAQ、プロフィール文案を作る

## 4. SNS観測レイヤー

今すぐSNS公式API連携に寄せすぎない。  
Instagram、TikTok、XはAPI制限や審査が重いため、初期は「URLを登録し、診断対象として管理する」構造にする。

このレイヤーは、いきなり本番機能として作るのではなく、将来の管理画面・有料診断・HCI研究へつながる「観測対象の入れ物」として先に設計しておく。

### 4.1 初期方針

今すぐ使うもの:

- Gemini
- Google Maps API
- 入力されたSNS URL
- 店主入力
- 画像があれば画像診断

まだやらないもの:

- Instagram API本格連携
- TikTok API本格連携
- X API本格連携
- SNS自動収集
- 投稿自動スクレイピング

### 4.2 将来の管理画面

店舗単位:

- 店舗ID
- 店名
- 地域
- 業種
- Google Maps URL
- Instagram URL
- TikTok URL
- YouTube URL
- X URL
- 公式サイトURL
- 診断対象ON/OFF

SNS投稿単位:

- 投稿URL
- 媒体
- 投稿日
- 投稿タイプ
- 診断対象ON/OFF
- 店舗ID
- タイトル
- 説明文
- 文字起こし
- サムネイル画像
- スクショ
- Gemini診断結果

### 4.3 動画診断軸

- Save: 保存したくなるか

### 4.5 SNS診断の扱い方

初期は、SNSを「自動収集する対象」ではなく、「店主または運営者が診断対象として登録する素材」として扱う。

理由:

- SNS公式APIは媒体ごとに制約が大きい
- 店舗ごとに本当に見たい投稿は限られる
- 研究用途では、どの投稿を観測対象にしたかを明示する必要がある
- 診断精度を上げるには、URLだけでなく、スクショ、サムネイル、文字起こし、運営者メモが重要になる

管理画面で持つべき状態:

- SNSアカウント単位の診断対象ON/OFF
- 投稿・動画単位の診断対象ON/OFF
- 診断済み / 未診断
- 店舗公開用に使う / 研究用だけに残す
- 有料診断に含める / 無料診断には含めない

この設計にしておくことで、将来Instagram、TikTok、YouTube、Xのどれか一つだけを先に強化しても、全体構造を壊さずに拡張できる。

### 4.6 世界観カード観測レイヤー

世界観カードは、静的な診断結果だけではなく、将来的には「感性漂流レイヤー」として扱う。

目的:

> 世界観カードを通じて、どんな感性の人が、どの経路から漂着し、どの店舗・SNS・Mapsへ回遊したかを匿名・非監視型で観測する。

重要原則:

- 実名追跡をしない
- mixi型の監視足跡にしない
- 「匿名の気配」を扱う
- 個人ではなく「感性導線」を見る
- 店舗改善、地域回遊、世界観設計へ活かす
- 「コミュニティ」を強く打ち出しすぎず、ゆるい共鳴・漂流・気配・回遊として扱う

観測対象:

- カード閲覧
- 滞在時間
- 再訪
- Mapsクリック
- SNSクリック
- QR流入
- 他カード回遊
- 保存意図
- シェア
- 時間帯
- 流入SNS
- 地域傾向

このレイヤーの目的は、PV解析ではない。

> どんな空気感を求める人が、どんな順番で店を見ているかを観測する。

将来的な可能性:

- 感性回遊分析
- 地域回遊分析
- 世界観クラスタ
- 同系統店舗の発見
- 「静かな店を好む人」などの傾向分析
- ゆるい共鳴コミュニティの可能性
- 世界観推薦

行動段階:

| 段階 | 状態 |
|---|---|
| Save | 保存した |
| Plan | 行こうと思った |
| Impulse | 今行きたい |
| Drift | 世界観を漂流・回遊した |

従来のマーケティングは購買やクリックを見がちだが、お客様.COMでは「空気を辿る行動」も観測対象にする。  
これは、世界観カードを媒介に、人がどんな空気を求め、どんな順番で回遊し、何に安心し、何に共鳴したかを見るための基盤である。

### 4.7 世界観カード画像のオーナー選択

世界観カードβ版では、AI診断に基づく推奨カード画像を自動表示する。  
ただし、店舗オーナーが「この画像はうちの空気と少し違う」と感じる場合がある。

そのため将来的には、以下の2つを分けて扱う。

- AI推奨カード
- オーナー選択カード

目的:

> 診断結果の合理性を残しながら、オーナーが「これなら発信したい」と思えるカードへ近づける。

重要原則:

- AI診断結果は変更しない
- 表示カード画像だけをオーナーが選べるようにする
- 「本来の診断ではこのカードが推奨」と分かる余地を残す
- 36枚の既存カードから選べるようにする
- 将来的には、カード種別ごとに画像候補を増やす

保存候補:

- recommended_card_id
- selected_card_id
- selected_by_owner
- selected_at
- selection_reason
- card_image_version

表示方針:

- お客様渡し用カードでは、オーナー選択カードを優先表示する
- 店舗管理側では、AI推奨カードとオーナー選択カードの両方を確認できるようにする
- 診断精度の検証では、AI推奨カードと実際に選ばれたカードのズレも観測対象にする

このズレは失敗ではなく、店舗オーナーの自己認識・発信したい空気・実際の店らしさを知る重要な材料になる。

- SNSリンクを開いた
- お店のリンク一覧を開いた
- 似た世界観カードへ移動した
- QRから来た
- 地域・コミュニティ系の導線に興味を示した

注意:

- 個人名、実名、アカウント名は表示しない
- 店舗側には個別ユーザーの追跡として見せない
- 「誰が来たか」ではなく「どんな空気の人が漂着したか」を見る
- 同意状態、匿名化、保存期間を後から設計できる余地を残す

店舗ダッシュボードでの表現案:

- 今日の漂着数
- よく見られた時間帯
- どのSNSから来たか
- Mapsを開いた人数
- SNSリンクを開いた人数
- 似た世界観カードとの回遊
- 「静かなカフェ系」「手仕事系」「地域交流系」などの感性傾向
- リピート閲覧の割合
- 世界観カードから来店前行動につながった割合

ユーザー側で出す場合の表現案:

- このカードに最近◯人が漂着しました
- この世界観を好きな人は、こちらのお店も見ています
- 静かな休日を探している人によく見られています
- 同じ空気感のお店

この機能は「監視」ではなく「匿名の共鳴観測」として設計する。  
個人を追いかけるのではなく、世界観カードを機縁にした感性導線、地域回遊、ゆるい共鳴コミュニティの可能性を見る。

## 9. 実装時の原則

### 9.1 AI診断は必ず構造化する

Geminiの出力は、できるだけ以下の構造へ寄せる。

- observation
- judgment
- evidence
- action

自由作文だけにしない。  
ただし、店舗オーナーに見せる画面では、硬いJSON感を出さず、やさしい日本語へ整える。

### 9.2 「なぜそう言えるか」を必ず残す

マジ診断に見えるかどうかは、点数よりも根拠で決まる。

悪い例:

> 入りやすさが弱いです。

良い例:

> 入口写真と駐車場写真が見当たらないため、初めて車で来る人は「どこに停めて、どこから入るか」で迷う可能性があります。

### 9.3 SEOは軽く、見え方は深く

SEOを捨てるのではなく、役割を限定する。

お客様.COMのSEO診断:

- 検索順位を上げる技術ではない
- 初めて検索した人が迷わず理解できるかを見る
- 地域名、業種、用途、営業時間、アクセス、価格感、FAQを軽く確認する

### 9.4 SNSは最初から研究に耐える形で持つ

SNS診断は、将来の論文・研究・地域マーケティングデータに接続できる可能性が高い。  
そのため、投稿単位で「どのAIモデル」「どのルーブリック」「どの診断バージョン」で見たかを残す。

### 9.5 店舗支援者と競合しない

お客様.COMは、経営コンサル、SEO会社、MEO会社、SNS運用代行と競合しすぎない。

立ち位置:

> 経営指導ではなく、初めて見る人からの「見え方診断」。

この立ち位置により、地域の支援者、デザイナー、写真家、SNS運用者、商工会、観光関係者とも組みやすくする。
